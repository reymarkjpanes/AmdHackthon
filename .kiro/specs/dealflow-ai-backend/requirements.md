# Requirements Document

## Introduction

DealFlow AI Backend is a FastAPI-based Python 3.11 service that powers the DealFlow AI enterprise document intelligence platform. It provides five REST API endpoints consumed by the existing Next.js frontend: document upload with text extraction and embedding generation, AI-powered multi-document analysis, RAG-based chat with evidence citations, PDF report export, and a pre-loaded demo data endpoint. The backend is designed for AMD MI300X GPU acceleration, uses Anthropic Claude as the primary LLM with a swappable AMD Developer Cloud provider, and is deployed to Railway.app.

The frontend is fully built at `dealflow-ai-frontend/` and its API calls define the exact request/response contract this backend must satisfy. All JSON field names, types, and structures in this document are normative â€” deviating from them will break the frontend.

## Glossary

- **Backend**: The DealFlow AI FastAPI service being specified in this document
- **Frontend**: The existing Next.js application at `dealflow-ai-frontend/` that calls the Backend
- **Session**: A server-side context identified by a UUID `session_id` that groups uploaded documents, their embeddings, and analysis results
- **Document**: A PDF, PNG, JPG, or JPEG file uploaded by the user, stored as parsed text chunks in the vector store for the duration of the Session
- **Chunk**: A contiguous segment of text extracted from a Document, approximately 512 tokens, stored as a vector embedding in ChromaDB
- **Embedding**: A 384-dimensional float vector produced by the `all-MiniLM-L6-v2` sentence-transformers model representing the semantic content of a Chunk
- **Vector_Store**: A ChromaDB in-memory collection scoped to a Session that stores Chunks alongside their Embeddings
- **LLM_Service**: The abstraction layer that routes LLM calls to either the Claude provider or the AMD_Provider
- **Claude_Provider**: The Anthropic `claude-3-5-sonnet` API, used as the primary LLM provider
- **AMD_Provider**: The AMD Developer Cloud inference endpoint stub, intended as a future drop-in replacement for the Claude_Provider
- **Document_Parser**: The service that extracts plain text from uploaded files using PyMuPDF for PDFs and pytesseract + Pillow for images
- **Analysis**: The structured AI-generated output for a Session containing executiveSummary, risks, comparisonMatrix, conflicts, and recommendation
- **Conflict**: A detected contradiction between information in two different Documents within the same Session
- **RAG**: Retrieval-Augmented Generation â€” the technique of embedding a user question, retrieving the top-k most similar Chunks from the Vector_Store, and passing those Chunks as context to the LLM
- **Report**: A PDF file generated by ReportLab containing AMD-branded Analysis results for a Session
- **Demo_Data**: Pre-computed static data returned by GET /api/demo representing five procurement documents and their Analysis, used for frontend demonstration without real uploads
- **ProcessingStatus**: One of the enum values: `uploading`, `uploaded`, `extracting_text`, `running_ocr`, `generating_embeddings`, `indexing`, `completed`, `failed`
- **Risk_Level**: One of the enum values: `HIGH`, `MEDIUM`, `LOW`
- **Severity**: One of the enum values: `HIGH`, `MEDIUM`, `LOW`

## Requirements

### Requirement 1: Document Upload and Processing Pipeline

**User Story:** As a frontend client, I want to POST multipart files to `/api/upload` and receive a session identifier plus per-document processing status, so that I can track processing progress and subsequently trigger analysis.

#### Acceptance Criteria

1. THE Backend SHALL expose a `POST /api/upload` endpoint that accepts `multipart/form-data` with a field named `files` containing one or more file uploads
2. THE Backend SHALL accept files with MIME types `application/pdf`, `image/png`, `image/jpeg` and reject all other MIME types
3. THE Backend SHALL accept between 1 and 10 files per request; IF a request contains more than 10 files, THEN THE Backend SHALL return HTTP 400 with `{"error": "You can upload up to 10 files at a time.", "code": "FILE_COUNT_EXCEEDED"}`
4. THE Backend SHALL enforce a maximum file size of 10,485,760 bytes (10 MB) per file; IF any file exceeds this size, THEN THE Backend SHALL return HTTP 400 with `{"error": "This file exceeds the 10 MB limit.", "code": "FILE_SIZE_EXCEEDED"}`
5. WHEN a valid upload request is received, THE Backend SHALL generate a UUID v4 `session_id` and create a new Session in the Vector_Store
6. WHEN processing a PDF file, THE Document_Parser SHALL use PyMuPDF (`fitz`) to extract plain text from all pages of the file
7. WHEN processing a PNG, JPG, or JPEG file, THE Document_Parser SHALL use pytesseract with Pillow to perform OCR and extract plain text from the image
8. THE Backend SHALL split extracted text into Chunks of approximately 512 tokens, with a 50-token overlap between consecutive Chunks to preserve context across boundaries
9. THE Backend SHALL generate an Embedding for each Chunk using the `sentence-transformers` model `all-MiniLM-L6-v2`
10. THE Backend SHALL store all Chunks and their Embeddings in a ChromaDB in-memory collection keyed by `session_id`
11. WHEN all files are processed successfully, THE Backend SHALL return HTTP 200 with a JSON body matching the `UploadResponse` shape: `{"sessionId": "<uuid>", "documents": [<UploadedDocument>, ...], "message": "<string>"}`
12. EACH `UploadedDocument` object in the response SHALL contain: `id` (string UUID), `filename` (original filename), `fileType` (`"pdf"` or `"image"`), `fileSize` (integer bytes), `uploadedAt` (ISO 8601 datetime string), `processingStatus` (`"completed"` on success or `"failed"` on per-file error)
13. IF text extraction fails for an individual file, THEN THE Backend SHALL set that document's `processingStatus` to `"failed"` and include it in the response rather than returning an error for the entire batch
14. IF all files fail processing, THEN THE Backend SHALL return HTTP 422 with `{"error": "All documents failed to process.", "code": "UPLOAD_FAILED"}`

### Requirement 2: Multi-Document AI Analysis

**User Story:** As a frontend client, I want to POST a `session_id` to `/api/analyze` and receive a complete structured Analysis, so that I can display executive summary, risks, comparison matrix, conflict detection, and a recommendation to the user.

#### Acceptance Criteria

1. THE Backend SHALL expose a `POST /api/analyze` endpoint that accepts a JSON body `{"sessionId": "<string>"}`
2. IF the `sessionId` does not correspond to an active Session, THEN THE Backend SHALL return HTTP 404 with `{"error": "Session not found.", "code": "SESSION_NOT_FOUND"}`
3. WHEN a valid `sessionId` is received, THE Backend SHALL retrieve all Chunks associated with that Session from the Vector_Store
4. THE Backend SHALL call the LLM_Service to generate an `executiveSummary` string describing the key findings across all documents in the Session
5. THE Backend SHALL call the LLM_Service to generate a `risks` array; EACH risk object SHALL contain: `id` (string), `level` (`"HIGH"`, `"MEDIUM"`, or `"LOW"`), `description` (string), `sourceDocument` (filename string), `category` (string)
6. THE Backend SHALL call the LLM_Service to generate a `comparisonMatrix` array; EACH row SHALL contain: `field` (string), `values` (object mapping document names to value strings), `winner` (optional string â€” the document name with the most favorable value for that field)
7. THE Backend SHALL invoke the Conflict_Engine to detect cross-document contradictions; EACH conflict object SHALL contain: `id` (string), `type` (string), `severity` (`"HIGH"`, `"MEDIUM"`, or `"LOW"`), `documentA` (`{"name": string, "excerpt": string}`), `documentB` (`{"name": string, "excerpt": string}`), `explanation` (string), `recommendedAction` (string)
8. THE Backend SHALL call the LLM_Service to generate a `recommendation` object containing: `title` (string), `summary` (string), `nextSteps` (array of strings), `confidence` (float between 0.0 and 1.0 inclusive)
9. WHEN all analysis steps (executive summary, risks, comparison matrix, conflict detection, and recommendation) have finished successfully, THE Backend SHALL return HTTP 200 with a JSON body matching the `AnalyzeResponse` shape: `{"sessionId": "<string>", "status": "completed", "analysis": <AnalysisResult>}`; THE Backend SHALL NOT return a response before all steps are complete
10. THE `AnalysisResult` SHALL contain: `analyzedAt` (ISO 8601 datetime string), `executiveSummary` (string), `risks` (array), `comparisonMatrix` (array), `conflicts` (array), `recommendation` (object)
11. IF the LLM_Service call fails, THEN THE Backend SHALL return HTTP 502 with `{"error": "Analysis failed. LLM service unavailable.", "code": "ANALYSIS_FAILED"}`
12. THE Backend SHALL store the completed `AnalysisResult` in the Session so it can be referenced by the report endpoint without recomputing

### Requirement 3: RAG-Based Chat with Evidence Citations

**User Story:** As a frontend client, I want to POST a `session_id` and `question` to `/api/chat` and receive a structured AI response with grounded evidence, so that I can display the answer, source citations, risks, and recommendation to the user.

#### Acceptance Criteria

1. THE Backend SHALL expose a `POST /api/chat` endpoint that accepts a JSON body `{"sessionId": "<string>", "question": "<string>"}`
2. IF the `sessionId` does not correspond to an active Session, THEN THE Backend SHALL return HTTP 404 with `{"error": "Session not found.", "code": "SESSION_NOT_FOUND"}`
3. IF the `question` field is absent or is an empty string, THEN THE Backend SHALL return HTTP 400 with `{"error": "Question is required.", "code": "INVALID_REQUEST"}`
4. WHEN a valid request is received, THE Backend SHALL generate an Embedding for the `question` using the same `all-MiniLM-L6-v2` model used during upload
5. THE Backend SHALL query the Session's Vector_Store to retrieve the top 5 Chunks whose Embeddings are most semantically similar to the question Embedding
6. THE Backend SHALL construct a prompt that includes the retrieved Chunks as context and instructs the LLM_Service to respond with a structured JSON containing: `answer` (string), `evidence` (array of evidence objects), `risks` (string), `recommendation` (string)
7. EACH evidence object SHALL contain: `quote` (verbatim excerpt from a Chunk, max 200 characters), `sourceDocument` (original filename), `documentType` (`"pdf"` or `"image"`)
8. WHEN the LLM_Service returns a response, THE Backend SHALL return HTTP 200 with a JSON body matching the `ChatResponse` shape: `{"messageId": "<uuid>", "role": "assistant", "structuredResponse": <StructuredAIResponse>, "processingTimeMs": <integer>}`
9. THE `StructuredAIResponse` SHALL contain: `answer` (string), `evidence` (array), `risks` (string), `recommendation` (string)
10. THE `processingTimeMs` field SHALL reflect the actual elapsed milliseconds from request receipt to response generation
11. IF the LLM_Service call fails, THEN THE Backend SHALL return HTTP 502 with `{"error": "Chat service unavailable.", "code": "ANALYSIS_FAILED"}`

### Requirement 4: PDF Report Generation

**User Story:** As a frontend client, I want to POST a `session_id` to `/api/report` and receive a binary PDF, so that users can download and share a professionally formatted analysis report.

#### Acceptance Criteria

1. THE Backend SHALL expose a `POST /api/report` endpoint that accepts a JSON body `{"sessionId": "<string>", "includeChat": <boolean>}`
2. IF the `sessionId` does not correspond to an active Session with a completed Analysis, THEN THE Backend SHALL return HTTP 404 with `{"error": "Session not found.", "code": "SESSION_NOT_FOUND"}`
3. WHEN a valid request is received, THE Backend SHALL generate a PDF using ReportLab containing the following sections in order: title page with DealFlow AI branding, executive summary, risk analysis table, comparison matrix table, detected conflicts, and recommendation
4. THE PDF title page SHALL include the text "DealFlow AI â€” Document Intelligence Report", the generation timestamp, and the text "Powered by AMD MI300X"
5. THE PDF SHALL use AMD brand color `#ED1C24` for section headers and accent elements
6. EACH risk in the PDF SHALL display the risk level (`HIGH`, `MEDIUM`, `LOW`), description, source document, and category
7. THE comparison matrix SHALL be rendered as a table with document names as column headers and comparison fields as row labels
8. EACH conflict entry SHALL display: conflict type, severity, affected document names, conflicting excerpts, explanation, and recommended action
9. THE recommendation section SHALL display the title, summary, next steps as a numbered list, and confidence score formatted as a percentage
10. THE Backend SHALL return the PDF as HTTP 200 with `Content-Type: application/pdf` and `Content-Disposition: attachment; filename="dealflow-report-<YYYY-MM-DD>.pdf"`
11. IF PDF generation fails, THEN THE Backend SHALL return HTTP 500 with `{"error": "Report generation failed.", "code": "REPORT_FAILED"}`

### Requirement 5: Demo Data Endpoint

**User Story:** As a frontend client, I want to GET `/api/demo` and receive pre-loaded procurement document data matching the `DemoData` type, so that the demo page can display a complete, realistic example without requiring real uploads.

#### Acceptance Criteria

1. THE Backend SHALL expose a `GET /api/demo` endpoint that requires no authentication and accepts no parameters
2. THE Backend SHALL return HTTP 200 with a JSON body matching the `DemoData` shape: `{"sessionId": "<string>", "documents": [<UploadedDocument>, ...], "analysis": <AnalysisResult>, "preSeededMessages": [<ChatMessage>, ...]}`
3. THE `documents` array SHALL contain exactly 5 `UploadedDocument` objects representing: `Supplier_A_Quotation.pdf`, `Supplier_B_Quotation.pdf`, `Existing_Contract_TechCorp.pdf`, `Invoice_TechCorp_March2026.pdf`, and `Company_Procurement_Policy.pdf`; ALL documents SHALL have `processingStatus: "completed"`
4. THE `analysis` object SHALL contain a realistic `executiveSummary` describing a price discrepancy between TechCorp's invoice ($48,500) and quotation ($45,200) and an upcoming contract renewal deadline
5. THE `analysis.risks` array SHALL contain at least 5 risk items including: one HIGH risk for the invoice overcharge, one HIGH risk for the contract renewal deadline, at least one MEDIUM risk (required regardless of how many HIGH risks are present), and at least one LOW risk
6. THE `analysis.comparisonMatrix` SHALL contain at least 5 rows comparing Supplier A, Supplier B, and TechCorp across fields including Total Price, Payment Terms, Delivery Time, Warranty, and Support SLA; Supplier B SHALL be the `winner` for at least 3 fields
7. THE `analysis.conflicts` array SHALL contain at least 2 conflict objects; the first SHALL be a HIGH-severity price discrepancy between `Invoice_TechCorp_March2026.pdf` and `Existing_Contract_TechCorp.pdf`
8. THE `analysis.recommendation` SHALL recommend Supplier B with `confidence` of 0.87, a `nextSteps` array of at least 4 items, and a `title` containing "Supplier B"
9. THE `preSeededMessages` array SHALL contain at least 4 `ChatMessage` objects (2 user + 2 assistant turns) with realistic procurement questions and `StructuredAIResponse` objects on assistant messages
10. ALL datetime fields in the demo response SHALL be valid ISO 8601 strings; the Backend SHALL not return JavaScript `Date` objects

### Requirement 6: LLM Provider Abstraction

**User Story:** As a developer, I want a `LLMService` abstraction class with a configurable provider enum, so that swapping from Anthropic Claude to AMD Developer Cloud requires only a single environment variable change.

#### Acceptance Criteria

1. THE LLM_Service SHALL implement a provider enum with at least two values: `CLAUDE` and `AMD`
2. THE LLM_Service SHALL read the active provider from the environment variable `LLM_PROVIDER`; IF `LLM_PROVIDER` is not set, THE LLM_Service SHALL default to `CLAUDE`; THE LLM_Service SHALL not validate that the environment value matches the runtime behavior of the active implementation
3. WHEN `LLM_PROVIDER` is `CLAUDE`, THE LLM_Service SHALL call the Anthropic API using model `claude-3-5-sonnet-20241022` and the key from `ANTHROPIC_API_KEY`
4. WHEN `LLM_PROVIDER` is `AMD`, THE LLM_Service SHALL call the AMD Developer Cloud endpoint from `AMD_CLOUD_ENDPOINT` using the key from `AMD_CLOUD_API_KEY` with Llama 3.2 Vision as the target model
5. THE LLM_Service SHALL expose a single async method `complete(system_prompt: str, user_prompt: str, max_tokens: int) -> str` that abstracts provider differences from callers
6. IF `ANTHROPIC_API_KEY` is not set and `LLM_PROVIDER` is `CLAUDE`, THEN THE LLM_Service SHALL raise a `ConfigurationError` at startup
7. THE AMD_Provider implementation SHALL be a fully functional stub that returns a mock response if `AMD_CLOUD_ENDPOINT` is not configured, enabling local development without AMD credentials; this mock behavior applies regardless of the active `LLM_PROVIDER` setting
8. THE LLM_Service SHALL log which provider is active at application startup

### Requirement 7: CORS and HTTP Configuration

**User Story:** As a frontend client running at a different origin, I want the backend to include correct CORS headers, so that browser requests from the Next.js dev server and production deployment are not blocked.

#### Acceptance Criteria

1. THE Backend SHALL configure FastAPI's `CORSMiddleware` to allow requests from all origins (`*`) in development and from the configured `ALLOWED_ORIGINS` environment variable in production
2. THE Backend SHALL allow HTTP methods: `GET`, `POST`, `OPTIONS`
3. THE Backend SHALL allow HTTP headers: `Content-Type`, `Authorization`, `X-Requested-With`
4. THE Backend SHALL respond to OPTIONS preflight requests with HTTP 200
5. THE Backend SHALL listen on host `0.0.0.0` and port `8000` by default, overridable via the `PORT` environment variable
6. ALL API endpoints SHALL be prefixed with `/api/`
7. THE Backend SHALL expose a `GET /health` endpoint that returns HTTP 200 with `{"status": "ok", "version": "<string>"}` for Railway deployment health checks

### Requirement 8: Error Response Format Consistency

**User Story:** As a frontend client, I want all error responses to follow a consistent JSON structure matching the `ErrorResponse` type, so that the API client's error normalization logic works correctly for all failure modes.

#### Acceptance Criteria

1. ALL error responses from THE Backend SHALL use the JSON shape `{"error": "<human-readable message>", "code": "<ERROR_CODE_STRING>", "details": <optional object or null>}`
2. THE Backend SHALL use the following error codes matching the frontend's `API_ERROR_CODES` constants: `FILE_SIZE_EXCEEDED`, `FILE_COUNT_EXCEEDED`, `UNSUPPORTED_FORMAT`, `UPLOAD_FAILED`, `ANALYSIS_FAILED`, `SESSION_NOT_FOUND`, `NETWORK_ERROR`, `UNAUTHORIZED`, `UNKNOWN_ERROR`
3. THE Backend SHALL return HTTP 400 for client input errors (invalid file type, file too large, too many files, missing required fields)
4. THE Backend SHALL return HTTP 404 for unknown session IDs
5. THE Backend SHALL return HTTP 422 for validation errors on request body schemas
6. THE Backend SHALL return HTTP 500 or 502 for server-side failures (LLM errors, internal processing errors)
7. IF an unhandled exception occurs, THE Backend SHALL catch it with a global exception handler and return HTTP 500 with `{"error": "An unexpected error occurred.", "code": "UNKNOWN_ERROR"}`
8. THE Backend SHALL never expose Python stack traces or internal exception messages in production error responses

### Requirement 9: Session Lifecycle Management

**User Story:** As a developer, I want sessions to be isolated in-memory and cleaned up automatically, so that the backend remains stateless across restarts and does not accumulate memory indefinitely.

#### Acceptance Criteria

1. THE Vector_Store SHALL maintain one ChromaDB in-memory collection per `session_id`, created at upload time and accessible throughout the Session lifecycle
2. THE Backend SHALL store Session metadata (document list, analysis results, file type mappings) in an in-memory Python dictionary keyed by `session_id`
3. WHEN a Session's ChromaDB collection is created, THE Backend SHALL name it using the `session_id` to ensure isolation between concurrent Sessions
4. THE Backend SHALL support at least 10 concurrent Sessions in memory without cross-session data leakage
5. Sessions SHALL NOT be persisted to disk; restarting the Backend clears all Sessions
6. THE Backend SHALL not implement automatic Session expiry for v1; Sessions live until the process restarts
7. WHEN the same `session_id` is used across `/api/analyze`, `/api/chat`, and `/api/report`, THE Backend SHALL return data scoped exclusively to that Session's documents and embeddings

### Requirement 10: Dependency and Deployment Configuration

**User Story:** As a developer deploying to Railway.app, I want a complete `requirements.txt`, `.env.example`, and `Procfile` so that the service can be deployed with a single push.

#### Acceptance Criteria

1. THE Backend SHALL include a `requirements.txt` listing all Python dependencies with pinned versions including: `fastapi`, `uvicorn[standard]`, `python-multipart`, `pymupdf`, `pytesseract`, `Pillow`, `sentence-transformers`, `chromadb`, `anthropic`, `reportlab`, `python-dotenv`, and `pydantic`
2. THE Backend SHALL include a `.env.example` file documenting all required and optional environment variables: `ANTHROPIC_API_KEY`, `AMD_CLOUD_API_KEY`, `AMD_CLOUD_ENDPOINT`, `LLM_PROVIDER`, `ALLOWED_ORIGINS`, `PORT`
3. THE Backend SHALL include a `Procfile` with the entry: `web: uvicorn main:app --host 0.0.0.0 --port $PORT`
4. THE Backend project root SHALL be `backend/` within the workspace, matching the folder structure specified in the implementation brief
5. THE Backend SHALL be compatible with Python 3.11 and SHALL NOT use features exclusive to Python 3.12 or later
6. THE Backend's `main.py` SHALL instantiate the FastAPI app, register all routers from `routers/`, configure CORS middleware, and register the global exception handler
7. THE Backend SHALL include a `tests/` directory with at least three test files: `test_upload.py`, `test_analysis.py`, and `test_conflict.py`

### Requirement 11: Document Parsing Round-Trip Integrity

**User Story:** As a developer, I want extracted text from documents to be verifiably complete and correctly chunked, so that the analysis and chat endpoints produce grounded, non-hallucinated results.

#### Acceptance Criteria

1. THE Document_Parser SHALL extract all readable text from each page of a PDF; for a PDF with N pages, the concatenated extracted text SHALL contain characters from every page that contains text
2. WHEN a PDF page contains no extractable text (e.g., a scanned image page), THE Document_Parser SHALL fall back to pytesseract OCR for that page
3. THE chunking algorithm SHALL produce Chunks such that reassembling all Chunks for a document (accounting for overlaps) yields a text that is a superset of the original extracted text
4. FOR ALL documents uploaded in a Session, the total number of Chunks stored in the Vector_Store SHALL equal the sum of chunks produced per document
5. THE embedding dimension for all Chunks SHALL be 384, matching the `all-MiniLM-L6-v2` model output dimension
6. WHEN the same text string is embedded twice, THE Embedding_Service SHALL produce identical float vectors (deterministic embedding)

### Requirement 12: Structured Prompt Engineering

**User Story:** As a developer, I want well-structured prompts in `prompts/` that produce consistently parseable LLM output, so that the analysis pipeline can reliably deserialize LLM responses into Pydantic models.

#### Acceptance Criteria

1. THE Backend SHALL store all LLM prompt templates as Python modules in the `prompts/` directory: `system_prompt.py`, `executive_summary.py`, `risk_analysis.py`, `conflict_detection.py`, `chat_copilot.py`, and `recommendation.py`
2. ALL prompts that require structured output SHALL instruct the LLM to respond exclusively in valid JSON matching a specified schema
3. THE risk analysis prompt SHALL instruct the LLM to classify each risk with exactly one of: `HIGH`, `MEDIUM`, or `LOW` for the `level` field
4. THE conflict detection prompt SHALL instruct the LLM to compare documents pairwise and identify only factual contradictions â€” not stylistic differences
5. THE chat copilot prompt SHALL instruct the LLM to ground every claim in a verbatim quote from the provided context Chunks, with `sourceDocument` set to the exact filename
6. THE Backend SHALL parse LLM JSON responses using Pydantic models; IF parsing fails due to malformed JSON, THE Backend SHALL retry the LLM call once before returning an error
7. THE system prompt SHALL identify the assistant as a procurement intelligence analyst powered by AMD MI300X hardware
